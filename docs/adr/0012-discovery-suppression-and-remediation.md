# 0012 — Discovery, suppression, deletion planning, and safe remediation

Date: 2026-07-19
Status: Accepted (founder interview, 2026-07-19)

## Context
The first full review against a private reference codebase exposed four related product gaps rather than isolated recognizer bugs:

- files ignored by Git were still eligible for discovery;
- the config `ignore` field made files invisible to the graph, so suppressing a claim could erase reference evidence and create false positives elsewhere;
- file claims covered only zero-inbound files, hiding unreachable groups whose members referenced one another;
- the product explained findings but could neither model their deletion consequences nor apply conservative, reviewable fixes.

The original PRD described v1 as detection-only. Before v0.1.0, the founder explicitly chose the conventional analysis-tool workflow: configuration and dry-run explanation first, then an opt-in `--fix` which mutates the working tree but never commits. Knip's [entry discovery](https://knip.dev/explanations/entry-files), [project-file guidance](https://knip.dev/guides/configuring-project-files), and [plugin model](https://knip.dev/explanations/plugins) were considered as references: configure roots and graph boundaries first, keep issue suppression distinct, and prefer convention-aware discovery over accumulating ignores. `unused` retains its stronger evidence, confidence, and auditability contracts.

## Decision

### Discovery and scope

- Discovery obeys the repository's applicable `.gitignore` files by default, including nested rules and negations. `--no-gitignore` is the explicit audit escape hatch.
- `project` defines which discovered files are eligible for claims. A project-excluded file may still be parsed when needed to preserve reference edges into claimable code.
- The old `ignore` graph-invisibility behavior is removed before release. It is not a compatibility promise because no stable release has shipped it.
- Convention and config-reference discovery remain the preferred way to describe liveness. Suppression is a policy overlay, not a substitute for a missing root or preset.

### Liveness and suppression

- A file is `unused` when it is unreachable from every production, configuration, and test root. Inbound references from other unreachable files do not make it alive.
- Existing declaration-level `/* unused:ignore <reason> */` directives remain supported.
- File-level and broader policy uses structured project/workspace config rules:

  ```jsonc
  {
    "suppressions": [
      {
        "files": ["public/generated/**"],
        "kinds": ["file"],
        "reason": "Generated by an external release job"
      }
    ]
  }
  ```

  `files`, at least one explicit claim `kind`, and a non-empty `reason` are mandatory. Rules are workspace-relative inside workspace overrides.
- Suppression never removes nodes or edges. Suppressed claims remain in canonical JSON and SARIF with their source, pattern, and reason; they are excluded from gates and default terminal listings, visible with `--show-suppressed`, and counted separately. A rule matching no claim emits an audit warning so stale policy cannot silently accumulate.

### Deletion consequences

- Consequences are counterfactual plans, not new claim verdicts. They do not affect baselines, gates, badges, or confidence.
- `unused why --delete <subject>` returns the selected subject, required re-export edits, and the claims that would become newly dead after the selected removal. The terminal groups consequences in deterministic stages; machine output uses the same structured plan.
- A single-subject plan is unsupported when any ordinary inbound reference from another file would remain, even when that caller is itself unreachable. Only re-export sites have a proven coordinated rewrite in v0.1; ordinary TypeScript, Elixir, Rust, and cross-language references cannot be advertised as safe to leave dangling. The planner and `--fix` share this preflight and report one deterministic stored source site.
- Deleting a separately reviewed atomic cohort of dead callers and targets may require no source edits, but cohort deletion is not represented by the schema-1.4 single-subject contract. Already-dead callers are prerequisites, not “newly dead” consequences, so they are never placed misleadingly into consequence stages. A future cohort shape requires a schema minor bump and separate founder approval.
- Reports may surface consequence summaries for their highest-value findings, but the claim remains the unit of truth. Plans are computed from the captured graph and claim run, never by reparsing for presentation.
- The canonical machine contract gains additive optional suppression provenance and deletion-plan shapes, requiring a schema minor bump under ADR 0006.

### Mutation

- `unused --fix` mutates the working tree and never commits, stages, installs packages, or contacts a network service.
- The eligible set is fixed from the initial analysis: only unsuppressed, high-confidence claims with verdict `unused`. `test-only`, medium, and low claims are never auto-fixed in v0.1.0.
- Default fix types are exports and dependencies. `--fix-type exports,dependencies,files` narrows the requested types. File deletion additionally requires `--allow-remove-files`; naming `files` without it is a usage error.
- Export fixes remove public exposure while preserving locally used declarations. Re-export specifiers are removed or rewritten as required. Unsupported or ambiguous syntax is skipped with a reason rather than guessed.
- A source export or file with any ordinary inbound reference from another file is skipped in v0.1.0. Only re-export edits have an automatic coordinated rewrite; the tool does not leave a now-invalid import in unreachable, excluded, or suppressed code.
- Every primary mutation and its required re-export edits are preflighted and applied as one rollback-capable transaction. Replacement files are staged beside their targets, installed atomically with original modes preserved, and restored if a later operation fails.
- Dependency fixes edit only the declaring manifest. Lockfiles and installed modules are deliberately left for the user's package manager.
- File fixes remove only eligible files selected by the initial analysis. Consequences discovered afterward are reported, not recursively deleted in a hidden fixpoint; a user may review and run the tool again.
- After mutation, the analyzer runs again and prints applied, skipped, and remaining counts. Suppressed claims are never mutated.
- `check`, `baseline`, `report`, `badge`, JSON/SARIF-only operation, and MCP remain read-only. MCP receives no mutation tool.

## Consequences

- Configuration can no longer manufacture false positives by erasing graph evidence.
- Complete reachability improves recall and makes dead strongly connected components visible; corpus labels, not prior output counts, decide whether the new claims are correct.
- The claim schema moves from 1.1 to 1.2 additively. Existing consumers that tolerate optional fields remain compatible.
- `--fix` is intentionally conservative and may skip valid cleanups. Precision and reviewability outrank mutation coverage.
- Destructive file removal has two explicit opt-ins and is always recoverable through the user's version-control workflow; `unused` itself makes no commit.
