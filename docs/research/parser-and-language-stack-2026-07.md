# Parser and language stack — research for Phase 2 ADRs (parser strategy, CLI runtime)

Research date: 2026-07-18. Facts verified live against primary sources (GitHub repo contents/API, npm/PyPI registries, official vendor blogs) on 2026-07-18 unless otherwise dated. Inference is flagged as such; everything else is a checked fact.

## 1. TS/JS analysis stack

**Knip (webpro-nl/knip, v6.27.0, `packages/knip/package.json` fetched live)**: runtime `dependencies` are `oxc-parser ^0.140.0`, `oxc-resolver 11.24.2`, `get-tsconfig 4.14.0`, plus `fdir`/`tinyglobby`/`picomatch` (file walking), `jiti` (config loading), `zod`, `yaml`, `smol-toml`. **No `typescript` package appears anywhere in dependencies or devDependencies** except `@typescript/native-preview` (tsgo, the Go-native compiler preview) as a devDependency used only in `"build": "tsgo"` — i.e. to compile Knip's own source, not to analyze user code. This confirms Knip v6 has **fully dropped the TypeScript compiler API**. Per Knip's own release post ([knip.dev/blog/knip-v6](https://knip.dev/blog/knip-v6), 2026-03-20): v6 replaced the old "TypeScript backend," which "carried the overhead of wiring up an entire program along with the typechecker," with oxc-parser for parsing and oxc-resolver for module resolution; `get-tsconfig` (not tsc) reads tsconfig paths/extends. Rationale stated explicitly: TS's own compiler APIs were "designed for IDEs and language servers," not one-shot static analysis, and "TypeScript itself [is] Go-ing places" (i.e., TS7/tsgo has no stable embedding API — see below) — replacing the backend "was only a matter of time." Measured perf gain: 2–4x across sampled repos (Astro 4.0s→2.0s, Rolldown 3.7s→1.7s, Sentry 11.0s→4.0s, TypeScript-on-itself 3.7s→0.9s). Crucially, Knip does **not** rely on oxc's full semantic/scope-resolution layer: oxc-parser's JS API returns AST + comments + an ESM `module` field (`staticImports`/`staticExports`/`dynamicImports`/`importMetas`) directly, with only optional cheap semantic *error* checking (`showSemanticErrors`) — no symbol table or scope tree is exposed over NAPI ([oxc.rs parser docs](https://oxc.rs/docs/guide/usage/parser), verified 2026-07-18). Knip builds its own reference graph on top of that import/export surface plus oxc-resolver, i.e. a **custom resolver/graph layer**, not a borrowed semantic-analysis engine.

**Fallow (fallow-rs/fallow, v3.6.0, `Cargo.toml` fetched live, repo pushed 2026-07-18, 4,130 stars)**: workspace deps pin `oxc_allocator`/`oxc_ast`/`oxc_ast_visit`/`oxc_parser`/`oxc_span`/`oxc_syntax` at `0.126`, `oxc_resolver` at `11`, and — unlike Knip — **`oxc_semantic 0.126`**, consumed as a native Rust crate rather than through NAPI. So Fallow uses OxC's full binding/scope-resolution layer directly in Rust, plus `lightningcss` for CSS/CSS-in-JS extraction. Its README states explicitly: "no TypeScript compiler or Node.js runtime needed for static analysis," and its own docs (`docs.fallow.tools/analysis/limitations`) self-disclose **"syntactic analysis, no type resolution"** — oxc gives Fallow syntax + binding/scope info, not full type inference.

**TypeScript compiler API / TS7 (tsgo)**: TS7 ("Corsa"), Microsoft's Go-native rewrite, GA'd **2026-07-08** ([announcement](https://devblogs.microsoft.com/typescript/announcing-typescript-7-0/)) — 10 days before this research date — claiming 8–12x full-build speedups (VSCode 125.7s→10.6s; Sentry 139.8s→15.7s). **Load-bearing risk fact: TS 7.0 ships with no stable programmatic API.** The announcement states tool authors "can only currently rely on TypeScript 6.0" for embedding; a "new (and different)" API is planned for 7.1, on the historical 3–4 month cadence (roughly October–November 2026, not firm). Tools that embed TS (Volar/Vue, Svelte, Astro, MDX) are told explicitly to wait. The interim shape is IPC via `libsyncrpc` (Rust NAPI), still "early days of API design" per the original preview post (2025-05-22). Net: the legacy JS-based TS 5.x/6.x compiler API is the only mature embeddable option today — but it's the "last JS-based release" per Knip's own post, a legacy target, not a forward one.

**oxc-parser npm bindings** (v0.140.0, published 2026-07-14, verified via npm registry): AST-only-plus-ESM-metadata surface as described above; no symbol/scope table over the JS boundary. **oxc-resolver npm bindings** (v11.24.2, published 2026-07-12): a Rust port of `enhanced-resolve` + `tsconfig-paths-webpack-plugin` + `tsconfck`; handles tsconfig `paths`/`extends` auto-discovery, package.json `exports`/`imports` conditional resolution (ESM/CJS), and has monorepo (yarn/pnpm workspace) test fixtures — this is the module-resolution layer both Knip and Fallow build on.

**swc** (`@swc/core` v1.15.43, published 2026-06-22): mature, widely embedded in framework toolchains (Next.js, etc.) for transforms, but oxc's own parser benchmark ([oxc-project/bench-javascript-parser-written-in-rust](https://github.com/oxc-project/bench-javascript-parser-written-in-rust)) reports oxc's parser ≥3x faster than swc and ~5x faster than Biome. Current guidance from third-party comparisons (2026): pick oxc for linting/analysis-style single-pass work, swc where a framework's build pipeline already depends on it. swc has no equivalent to a public, well-documented `oxc_semantic`-style scope/symbol crate that either Knip or Fallow chose to adopt.

## 2. tree-sitter suitability

GitHub's **stack-graphs** (github/stack-graphs) — the highest-profile production attempt at tree-sitter-based, incremental name resolution at scale — is **archived** (owner-archived 2025-09-09, confirmed via GitHub API `archived: true`, 874 stars, 0 open issues). No successor project or stated reason was found in the repo. Tree-sitter itself remains widely used for syntax highlighting, structural search (ast-grep), and lightweight code-intelligence/knowledge-graph tools (e.g. a February 2026 arXiv project, `tree-sitter-analyzer`), but these build call/reference graphs from syntax alone — cross-file, package.json/tsconfig-aware module resolution is out of scope or bolted on ad hoc. No production dead-code/reference-graph tool using tree-sitter as the resolution layer (not just the parser) was found for TS/JS. **Verdict: not production-suitable as a resolution engine today** — it lacks type/module-graph awareness by design, and its most serious attempt at closing that gap is now archived.

## 3. Elixir

`mix xref` ships in Elixir/Mix core and offers `graph` (dependency graph with `--format`/`--sink`/`--source` filters), `callers CALLEE` (Module, Module.function, or Module.function/arity), and `trace FILE` ([hexdocs.pm/mix/Mix.Tasks.Xref.html](https://hexdocs.pm/mix/Mix.Tasks.Xref.html)). `mix xref unreachable`/`deprecated` were folded into compiler warnings in recent Elixir versions. Separately, the compiler exposes a **tracer API** (`Code.put_compiler_option(:tracers, [...])`), where a tracer module's `trace/2` receives every compile-time reference event (`:on_module`, function calls, etc.) — a first-class, native hook for building a precise reference graph without parsing Elixir source at all. **No actively maintained standalone dead-code tool exists for Elixir**: `mix_unused` (hauleth/mix_unused, unused *public* functions) is the closest match but is stale — last pushed 2023-12-26, 229 stars, 9 open issues, no release since. `styler` (hex.pm) is a style/format tool, not liveness detection. **Feasibility**: shelling out to `mix xref`/compiler tracing is clearly the right approach over writing an Elixir parser — it reuses the real compiler's own resolution (macros, protocols included) rather than reimplementing it, which is unusually cheap and unusually accurate versus every JS-ecosystem case above.

## 4. Python (brief)

Vulture (v2.16, 2026-03-25) is AST-based (Python's `ast` module), assigns a 60–100% confidence score, and needs manual whitelists to suppress false positives from dynamic dispatch/decorators/ORM patterns — its core FP driver. `deadcode` (albertas/deadcode) claims broader rule coverage than vulture but is stale (PyPI latest 2.4.1, 2024-08-09) — not meaningfully "newer" in practice. Ruff covers only locally-unused code (unused imports/vars) fast, not cross-module liveness. For **monorepo-aware import resolution**, `grimp` (python-grimp/grimp) is the most credible library — actively maintained (v3.15, published 2026-07-03), provides a queryable `ImportGraph` API — versus `pydeps`/`modulegraph`, which are bytecode/opcode-based, older, and visualization-first rather than API-first.

## 5. MCP TypeScript SDK

`@modelcontextprotocol/typescript-sdk` is actively developed (not archived; last push 2026-07-18, 12,882 stars, 466 open issues) and `@modelcontextprotocol/sdk` on npm is at v1.29.0 (published 2026-03-30). Stdio transport (`StdioServerTransport`) is supported and documented as the recommended path for local-tool/CLI integrations (vs. Streamable HTTP for network-accessible servers). The SDK is mid-transition to a v2 aligned with the 2026-07-28 protocol revision; v1.x is explicitly the supported production line, guaranteed bug/security fixes for at least 6 months after v2 ships — a safe, low-risk dependency to embed today.
