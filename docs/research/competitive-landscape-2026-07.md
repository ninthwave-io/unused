# Competitive landscape — dead code / unused dependency detection

Research date: 2026-07-18. All facts verified live via npm/PyPI/crates.io/GitHub APIs, not training memory. Dates noted inline.

## Knip (github.com/webpro-nl/knip)

**Maintenance**: v6.27.0, published 2026-07-15; ships roughly every 2–3 days (610 versions total). 10.4M weekly / 43M monthly npm downloads. 11,762 GitHub stars. Last 3 months: 410 commits, 101 issues opened / 110 closed, only 6 open issues today — aggressive triage, not neglect. **83% of recent commits are from a single maintainer** (Lars Kappert / webpro) — real bus-factor risk. Funding is modest: Open Collective shows $4,430 total raised, 8 contributors, ~$13.7k/yr budget; recognizable GitHub Sponsor logos (Vercel, Sourcegraph, Datadog, CodeRabbit) but only ~$520/mo gross — not corporately funded.

**Detects**: 16 issue types (files, deps, devDeps, unlisted, binaries, pnpm catalog, unresolved imports, exports, types, namespace exports/types, enum members, duplicates, cycles). 177 framework plugins (Next.js, Astro, Remix, Vue/Nuxt, Svelte, Angular, NestJS) plus npm/yarn/pnpm workspace, Nx and Turborepo support.

**Interfaces**: reporters include `json`, `markdown`, `codeclimate`, `codeowners`, `github-actions` (PR annotations) — **no SARIF reporter exists**, and the GitHub Action is third-party, not first-party. Has an official `@knip/mcp` (v0.0.32, published 2026-06-16, first published 2025-12-17) but adoption is tiny: ~2,700–12,600 weekly downloads vs. 10.4M for core Knip (~0.03–0.1%).

**Real FP pain (all github.com/webpro-nl/knip/issues, checked 2026-07-18)**: dynamic imports/lazy loading — [#556](https://github.com/webpro-nl/knip/issues/556) (2024-03), [#839](https://github.com/webpro-nl/knip/issues/839) React.lazy (2024-11); config/entry-point friction — [#1723](https://github.com/webpro-nl/knip/issues/1723) Vite babel field, filed 2026-05 (still recurring in current version); Vue/Next "magic" — [#740](https://github.com/webpro-nl/knip/issues/740) Vue SFC namespace import (4 months to close, 12 comments), [#1466](https://github.com/webpro-nl/knip/issues/1466) Nx+Next.js Pages Router regression causing 361 false positives (2026-01→02); monorepo — [#1642](https://github.com/webpro-nl/knip/issues/1642) regression when run from a workspace subpackage (filed 2026-03-26), [#987](https://github.com/webpro-nl/knip/issues/987) pnpm catalog support took 8 months (2025-03→11), [#1885](https://github.com/webpro-nl/knip/issues/1885) pnpm `dlx`+catalog still open, filed 2026-07-15; general — [#589](https://github.com/webpro-nl/knip/issues/589) "False positives that I can't seem to make go away" and [#853](https://github.com/webpro-nl/knip/issues/853) package.json `exports` resolution (18 comments, 4.5 months open). Pattern: issues close fast on average, but the framework-magic and monorepo classes recur across years and versions.

## depcheck, ts-prune, unimported — all archived, all redirect to Knip

All three are GitHub-archived, carry a first-party README notice recommending migration to Knip, and have no first-party MCP server:

- **depcheck**: archived (last commit 2025-02-27, adding the deprecation notice). npm latest **1.4.7** (2023-10-17), 1.58M weekly downloads. Detects unused/missing deps only, has `--json`, no SARIF/CI action. Open FPs: [#885](https://github.com/depcheck/depcheck/issues/885) (2024-02), [#898](https://github.com/depcheck/depcheck/issues/898) Vite alias (2024-07). Unofficial third-party `depcheck-mcp` exists (0.1.2, 2026-05-31).
- **ts-prune**: archived 2025-09-19. npm latest **0.10.3** stale since 2021-12-12 despite a pre-archive commit burst. Exports-only, **no JSON output ever added** ([#150](https://github.com/nadeesha/ts-prune/issues/150)), no dynamic-import support ([#29](https://github.com/nadeesha/ts-prune/issues/29)), a point release once flagged everything unused ([#151](https://github.com/nadeesha/ts-prune/issues/151)).
- **unimported**: archived earliest, 2024-03-10, with 18 issues mass-closed same day. npm latest **1.31.1** (2023-11-18). Files+deps+unresolved-imports, production-mode only, Next/Vue/Meteor/RN presets. Recurring "all files unimported" bug across three versions: [#59](https://github.com/smeijer/unimported/issues/59), [#64](https://github.com/smeijer/unimported/issues/64), [#71](https://github.com/smeijer/unimported/issues/71).

## vulture (Python, github.com/jendrikseipp/vulture)

Active: v2.16 (2026-03-25), pushed 2026-04-30, 4,694 stars. AST-based, confidence score 60–100%, whitelist for FP suppression. No JSON/SARIF, no official MCP (third-party `mcp-server-analyzer`, 11 stars). Core pain is whitelist overhead against dynamic dispatch: [#109](https://github.com/jendrikseipp/vulture/issues/109) (21 comments), [#8](https://github.com/jendrikseipp/vulture/issues/8) overridden C-extension methods (19 comments).

## deptry (Python, now osprey-oss/deptry)

Very active: v0.25.1 (2026-03-18), pushed today (2026-07-18), 1,436 stars. Notably **moved from the original author's personal account to a multi-maintainer "Osprey OSS" org** — governance maturation signal. Detects unused/missing/transitive deps; JSON output, pre-commit hook, GitHub Action, and a new `# deptry: ignore` inline suppression (0.25.x) — a direct FP-fatigue mitigation. No SARIF, no MCP. Pain: import-name-vs-package-name mismatches, e.g. [#1089](https://github.com/osprey-oss/deptry/issues/1089) (open since 2025-03).

## cargo-machete (Rust)

Active: v0.9.2 (2026-04-15), pushed 2026-07-13, 1,337 stars. Deliberately "fast but imprecise" — scans source strings, no macro/feature-flag resolution, so it accepts false positives like [#78](https://github.com/bnjbvr/cargo-machete/issues/78) (open since 2023) as a tradeoff. No JSON/SARIF/MCP.

## New entrants, 2024–2026 — the picture has genuinely shifted

GitHub's `dead-code` topic has 106 repos, `unused-code` has 23, mostly created in 2025–2026. Most direct competitor: **Fallow** (fallow-rs/fallow, created 2026-03-17, 4,130 stars) — a Rust/OxC static analyzer for TS/JS with a **paid runtime tier ingesting V8/Istanbul production coverage for "cold-path deletion evidence"** — exactly `unused`'s static+runtime fusion thesis — plus an MCP server (`fallow-mcp`) and public speed benchmarks against Knip. **Reaper/ReaperAI** (Sentry-backed, getsentry/Reaper-iOS, OSS since 2025-07-08) is the credibility anchor for runtime-instrumented deletion (Duolingo case study) but is mobile-only. **necro** (manehorizons/necro, npm, ~1 star, June 2026) is conceptually the closest match to `unused`'s evidence-ladder design: three confidence tiers, per-finding evidence chains, and an explicit **"test-only" liveness verdict** — tiny and unproven, but proves independent convergence on this exact framing. **Skylos** (duriantaco/skylos, 474 stars, v4.29.0, 2026-07-09, weekly releases) covers 10 languages with confidence thresholds, JSON/SARIF, GitHub code-scanning, MCP server, and optional runtime tracing; its author's HN post (2026-02-03) argues explicitly for confidence-graded rather than binary verdicts. **repowise** (3,711 stars) bundles dead-code detection into a broader 9-tool MCP "codebase intelligence" platform. Knip's own v6 (2026-03-20, oxc rewrite, 2–4x faster) added no runtime/telemetry features — confirmed by direct roadmap/changelog check. No OTel/APM trace-correlation dead-code tool was found anywhere (checked via multiple targeted negative searches) — genuine open niche. No standalone product where an LLM performs the liveness judgment itself was found either; only an academic paper (DCE-LLM, arXiv 2506.11076, NAACL 2025, no shipping repo).

## Gaps & positioning

Nothing found combines all of: (1) graded confidence + provenance on every claim as a product-level guarantee (Knip, depcheck, vulture, deptry, cargo-machete all emit binary verdicts); (2) an explicit test-only liveness tier separating production-dead-but-test-alive code (only necro's early prototype attempts this); (3) production runtime/analytics correlation via OTel/APM/access-logs rather than in-process code coverage (Fallow and Reaper use coverage/instrumentation SDKs, not trace/log correlation — zero-integration log-based liveness is unclaimed); (4) zero-telemetry, local-first CLI plus a serious MCP server, done well — MCP is already a 6+-entrant space (Knip, necro, Skylos, repowise, weavatrix, assorted 0-star clones) but none combine it with #1–#3.

## npm naming findings

- **`unused`** (unscoped): v0.2.2 latest, confirmed abandoned. Created 2011-08-29 by Tomaz Muraus (github.com/Kami/node-unused); 0.2.2 itself published 2013-07-03; registry "modified" timestamp of 2022-06-28 is a metadata touch (maintainer list change), not a real release. GitHub repo not archived but last real commit 2019-01-23, 57 stars, 4 open issues. "A module which reports unused variables in your code."
- **`@unused` scope**: confirmed **claimed/registered** but dormant. `registry.npmjs.org/-/org/unused/user` returns `{"unused":"owner"}` (HTTP 200) — a control test against a random nonexistent scope correctly returns 404 "Scope not found," so this is a real positive signal, not an artifact. However `-/org/unused/package` returns `{}` and npm search for `scope:unused` returns 0 packages — zero packages ever published. `@unused/cli` itself 404s. `npmjs.com/org/unused` returned 403 to both curl and WebFetch, but a control fetch of a definitely-valid page (`npmjs.com/org/babel`) also 403'd — that's a blanket bot-block in this environment, not org-specific evidence; the registry API check above is the reliable source. **Net: `@unused/*` is not freely claimable** without contacting npm support about the dormant org.
- **`@unused.dev/cli`**: the `unused.dev` scope is **not registered** (`-/org/unused.dev/user` → 404 "Scope not found") — available if that org/user is created first.
- **`unused-dev`, `unusd`**: both 404 on the registry — available.
- **`unused-cli`**: 404 — confirmed available, consistent with prior knowledge.
