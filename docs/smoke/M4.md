### Dependency claims: 0 across all four repos — investigated, not a bug

T4.1 landed dependency claims this milestone, so this run's first job per
the task brief was checking them by hand. All four repos report **zero**
dependency-kind claims. Traced to ground truth two ways:

1. **Code path confirmed intentional.** `dependencies.ts`'s
   `isMaybeBinDependency` (lines 379–415) documents "pre-install
   conservatism": when a unit's `node_modules` doesn't exist at all (true
   for all four uninstalled clones), every declared dependency not already
   proven alive by an import/alias/hoisting edge is kept alive rather than
   claimed, since a bin-only CLI dependency can't be ruled out without an
   installed manifest.
2. **Confirmed empirically, not just by absence.** `npm install --omit=dev
   --ignore-scripts` on the axios clone (its 4 declared deps:
   `follow-redirects`, `form-data`, `https-proxy-agent`, `proxy-from-env`)
   still produced 0 dependency claims — because all 4 are genuinely
   imported in `lib/adapters/http.js` / `lib/core/Axios.js` / etc.
   (`grep` confirmed each). Re-running `unused --json` against the
   installed copy changed nothing. zod's per-package breakdown (below)
   independently confirms every workspace unit's declared deps were
   evaluated (claims land in every unit's own files, so the dependency
   pass definitely ran over all 8 units) — `packages/docs`'s 16 real deps
   (`next`, `react`, `fumadocs-*`, …) are all genuinely imported by its
   `app/`/`components/` tree.

**Verdict: zero dependency claims on this corpus is correct, not silent
failure.** These four repos are simply well-maintained — real popular OSS
libraries don't carry much dead `dependencies`. This is a coverage gap in
the *smoke methodology*, not the feature: the existing dependency-claim
*unit* tests (`dependencies.test.ts`) already cover the true-positive path
against synthetic fixtures with installed `node_modules`; this run adds
real-world confirmation of the *true-negative* path (no false claims on
real, healthy dependency graphs) but not a real-world true-positive. Noted
as recommended-fix #4 below.

### Worst finding: workspace-member `tsconfig.json#paths` are silently ignored in a monorepo

`packages/docs` (zod's Next.js/Fumadocs documentation site) declares its
own `tsconfig.json` with `"paths": { "@/*": ["./*"] }` — the extremely
common Next.js "`@/` maps to the app root" convention. Its real, working
Next.js App Router entrypoints (`app/(doc)/[[...slug]]/page.tsx`,
`app/layout.tsx`, `app/blog/layout.tsx`) are **correctly** recognized as
live by the `next` preset (T4.4) — none of them appear in the claim set.
But every file those entrypoints reach *only* through the `@/` alias —
`components/tabs.tsx`, `components/heading.tsx`,
`components/copy-markdown-button.tsx`, `loaders/get-llm-text.ts`,
`loaders/source.ts`, `app/layout.config.tsx`, `components/sidebar-item.tsx`,
`components/sidebar-logo.tsx`, `components/inkeep-search.tsx`,
`components/inkeep-bubble.tsx`, `loaders/stars.ts` (11+ confirmed by
grepping each file's real importer, e.g. `app/(doc)/[[...slug]]/page.tsx:3:
import { Tabs } from "@/components/tabs"`) — is flagged unused.

**Root cause, traced to source:** `analyze.ts:223` constructs exactly
**one** `Resolver` for the entire run, with `projectRoot: root` — the
*monorepo root*, not each workspace unit. Inside `Resolver`'s constructor
(`resolve.ts:304`), `getTsconfig(this.projectRoot)` therefore discovers
only zod's **root** `tsconfig.json` (which has no `paths` entry at all —
zod's own `packages/zod` unit doesn't need one). `packages/docs`'s own
`tsconfig.json#paths` is never read, for any file in that unit, for the
entire analysis. `readTsconfigOptionsForUnits(units)` (the very next line)
already reads *some* options per-unit (`emitDecoratorMetadata`, project
`references`) — module resolution/`paths` was not extended the same way.

**Reproduced minimally and in isolation**, to rule out anything else about
zod's specific shape: a 6-file pnpm workspace (`packages/app/tsconfig.json`
with `"paths": {"@/*": ["./*"]}`, an `index.ts` importing
`@/components/widget`) produces a **HIGH-confidence** false claim on
`widget.ts` — full HIGH, not hedged, because there's no coincidental
whole-package hazard cap in the minimal repro the way there is on real
zod. The identical fixture with the workspace wrapper removed (single
package, same tsconfig, same alias) resolves **correctly** — zero claims —
confirming the defect is specifically about workspace-member tsconfig
discovery, not path-alias resolution in general (the existing
`tsconfig-paths-alias`/`broken-paths-alias`/`workspace-pnpm-monorepo`
fixtures don't cover this exact combination: the workspace fixture's alias
points *cross-package* to a sibling, never a member's *own* root-mapped
`@/*`).

**This is this run's hono-shaped finding**: on real zod, it's masked to
medium by the same `unresolvable-entrypoint-target` cap covering the whole
repo (coincidence, not protection — a zod-shaped monorepo that happened to
ship a built `dist/` would fail the "zero high-confidence FP" gate on
every `@/`-aliased file in `packages/docs`, and the Next.js `@/` alias
convention is near-universal, not a zod idiosyncrasy). Recommended fix #1
below.

Separately (not a bug, a scope gap worth documenting): the remaining
`packages/docs` components with zero importers anywhere in `.ts`/`.tsx`
source (`platinum.tsx`, `featured.tsx`, `bronze.tsx`, `gold.tsx`,
`silver.tsx`, `ecosystem.tsx`, `hero-logo.tsx`, `accordion.tsx`,
`themed-image.tsx`, `codec-image.tsx`, `if.tsx`) are referenced from
`.mdx` content files (`content/ecosystem.mdx`, `content/codecs.mdx`,
`content/index.mdx`) as JSX-like tags via fumadocs' MDX-component-registry
convention — not a static import edge at all, and `.mdx` isn't a parsed
source type in v1. Genuinely out of the analyzer's current visibility, not
a false claim the analyzer should be expected to avoid yet; flagged as
future scope (recommendation #5).

## Performance checkpoint verdict (founder directive, M4 gate)

**CLAUDE.md's exact trigger: "cold runs >3× Knip, or clearly off the PRD
§8 trajectory, opens the ADR 0004 reversal (native/Rust hot paths)
immediately."**

**Verdict: do not open the ADR 0004 reversal. The data does not support it.**

- **Worst observed ratio this run: 1.23× (fastify)** — 41% of the way to
  the 3× trigger, in the wrong direction from M3's worst (1.10×, also
  fastify) but still comfortably sub-2× let alone sub-3×. Two of four
  repos (axios 0.64×, zod 0.92×) are *faster* than knip outright.
- **No repo across two milestones (7 total runs: 3 in M3 pre/post-fix + 4
  here) has ever exceeded 1.23×.** There is no trend line pointing toward
  3× — ratios this run are flat-to-improved versus M3, not degrading as
  the corpus grew larger and more structurally complex (a monorepo, more
  workspace units, dependency analysis, preset detection all added this
  milestone with no measurable per-file cost increase).
- **Wall-clock headroom against the actual PRD §8 ceiling is enormous** —
  every run this milestone completes in under 0.61% of the 60-second cold
  budget. Even a naive, uninvestigated 10× regression from here would
  still be inside budget at these file counts.
- **The honest caveat, unchanged from M3:** none of the four repos is
  within an order of magnitude of the ~5,000-module scale the PRD §8
  budget targets, so "off the PRD §8 trajectory" can't yet be assessed
  directly against the budget's own intended scale — only the *ratio to
  knip* (which the founder directive treats as the primary trigger) is
  fully assessable today, and that signal is unambiguous and healthy.
- **Recommendation:** keep the TypeScript/Node stack (ADR 0004 stands),
  continue the existing bench-every-milestone cadence, and treat "find or
  construct a genuinely ~5,000-module target repo" as a standing action
  for a future milestone (M5 or later) so the budget itself gets
  validated at scale before v1 ships — not as a performance-pivot
  trigger, but as a measurement-completeness gap.

## Recommended fixes, ranked

1. **Workspace-member `tsconfig.json#paths` are silently ignored in
   monorepo analysis** (this run's "Worst finding," zod/`packages/docs`).
   Confirmed root cause: `analyze.ts:223` builds one `Resolver` bound to
   the monorepo root; `Resolver`'s `getTsconfig()` call therefore never
   sees a workspace member's own `tsconfig.json#paths`. Reproduced at
   **HIGH** confidence in a minimal, isolated fixture (only masked on real
   zod by a coincidental whole-package hazard cap — the same "one hazard
   happens to save the gate" pattern M3's hono finding warned wouldn't
   always hold). The Next.js `@/`-root-alias convention this breaks is
   near-ubiquitous, not a zod idiosyncrasy — any Next.js/Vite app living
   inside a workspace member (increasingly the default project shape for
   both frameworks) hits this. Minimum fix: construct (or re-tsconfig) the
   `Resolver` per workspace unit, the same way `readTsconfigOptionsForUnits`
   already does for `emitDecoratorMetadata`/project `references`. Highest
   priority — this is a confirmed, reproducible, HIGH-confidence-capable
   false-positive class, exactly the category the top quality metric
   exists to catch before it reaches a real user whose monorepo doesn't
   happen to have a masking hazard.
2. **A regression fixture for the workspace-`paths` gap.** The existing
   `tsconfig-paths-alias` fixture aliases to a subdirectory
   (`@app/* → ./src/app/*`) at the package root; the existing
   `workspace-pnpm-monorepo` fixture's alias points *cross-package* to a
   sibling. Neither covers a workspace **member's own** root-mapped
   `@/* → ./*` alias — add one (mirrors the minimal repro built for this
   triage: a 2-package pnpm workspace, one member with its own
   `tsconfig.json#paths`, an import only reachable through that alias).
3. **Real-world true-positive coverage for dependency claims.** T4.1's
   dependency-claim mechanism checked out clean on all four repos (zero
   false claims, confirmed both by code trace and by an installed-copy
   re-run on axios), but that's entirely a true-*negative* result — these
   are well-maintained repos with no actually-dead `dependencies`. The
   existing unit tests already cover synthetic true-positive fixtures;
   what's missing is a real-world confirmation that the feature catches a
   genuinely unused dependency in an installed, real repo. Low urgency
   (no evidence of a defect) but worth a note for the next smoke round:
   deliberately pick or construct a target with an installed, known-dead
   `dependencies` entry.
4. **The residual per-export reachability inconsistencies** (hono's
   `EventV1Processor`/`EventV2Processor`/`ALBProcessor` vs
   `LatticeV2Processor`; zod's `benchmarks/index.ts` importing
   `datetime.ts` while only `index.ts` itself is claimed). Both are
   concrete, reproducible, narrow — plausibly the same underlying
   mechanism (file-level claims may require zero importers of any kind,
   while export-level claims follow strict entrypoint-BFS) surfacing
   twice on two unrelated repos. Worth a core-implementer trace with graph
   instrumentation rather than further black-box triage; low priority
   since in every observed case the *effect* is "under-claim" (a recall
   gap, safe direction) not "over-claim."
5. **Document, don't fix urgently: MDX-registered component liveness.**
   zod's `packages/docs` components referenced only from `.mdx` content
   (fumadocs' component-registry convention, not a static import) are
   correctly invisible to v1 (`.mdx` isn't a parsed source type) —
   genuinely out of scope, not a defect, but worth a documented assumption
   so it isn't a surprise on first run for the growing set of
   MDX-content-driven docs sites. Knip's own default behaviour here
   (deeper per-export analysis inside `ecosystem.tsx`, item 8 in the
   zod differential sample) suggests it may have MDX-aware tooling
   support worth studying per CLAUDE.md's "study the incumbents" directive
   before this becomes a priority.
6. **Lower priority, unchanged from M3:** npm-script/CI-workflow-invoked
   files and nested-sibling-package boundaries remain the largest single
   FALSE-POSITIVE volume by claim count across all four repos this run
   (all 20 hono file claims, all 21 fastify claims, ~24 of zod's 27
   ours-only claims once the alias bug is excluded) — but every instance
   this run continues to be correctly hedged at medium, never high, and
   the zod `packages/bench`/`packages/treeshake` differential shows
   **perfect agreement with knip** on this exact class (29/29 and 21/21
   files), reinforcing M3's conclusion that this is a real, shared,
   industry-wide gap rather than a defect unique to `unused`.

## Summary

- **Total high-confidence claims across all 4 repos: 0.** The M3 phasing
  acceptance line ("zero confirmed high-confidence FPs surviving the
  gate") holds — as it did in M3's post-fix state — with nothing new to
  hand-triage at high confidence, including zero dependency-kind high
  claims (investigated and confirmed correct, not silently absent).
- **But "zero high" is once again a hazard-cap artifact on 3 of 4 repos**,
  not proof of a defect-free analyzer: hono (tsconfig `references`),

  unbuilt entrypoints), fastify (`computed-require`). This run confirms a

  exists right now — the workspace-`paths` bug — masked on real zod by
  exactly this pattern, exactly as M3 warned wouldn't always hold. It is
  this milestone's highest-priority fix.
- **Zod, the M3-skipped monorepo, analyzes cleanly**: exit 0, all 7 real
  workspace units discovered and evaluated, no refusal needed. T4.2's
  workspace support handles a real-world pnpm monorepo on the first
  attempt — a genuine M4 milestone win, immediately followed by finding
  the milestone's most important defect specifically *because* a real
  monorepo was in the corpus for the first time.

  knip's degraded-config behaviour**, not just matches its intent — knip

  load failures on the uninstalled clones), while our zero-config
