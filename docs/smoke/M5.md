## Export/file `test-only` claims — the aws-lambda cluster (hono)

Sampling hono's 35 `export`/`test-only` claims (task requirement #2:
"every HIGH claim... a test-only claim on production-alive code is an
FP" — extended here to the medium population since high is empty) surfaced
a second, independent confirmed bug, in `src/adapter/aws-lambda/handler.ts`
— **the same file M4 flagged as an unresolved "residual anomaly"** (four
processor classes sharing an identical shape; only one claimed, three
silently alive, mechanism untraced).

**In M5 the anomaly has changed shape, not resolved:** 9 of the 35 claims
—`getProcessor`, `EventV1Processor`, `EventV2Processor`, `ALBProcessor`,
`APIGatewayProxyEvent`, `APIGatewayProxyEventV2`, `ALBProxyEvent`,
`LatticeProxyEventV2`, `isContentEncodingBinary`— are all one coherent
cluster, and all are **confirmed false positives** by direct code trace:

```ts
// handler.ts:239, confirmed alive (no claim, re-exported via index.ts, T3.6)
export const handle = <...>(...) => {
  // handler.ts:253 — a same-file call to another exported symbol
  const processor = getProcessor(event)
  ...
}
// handler.ts:624 — getProcessor is itself exported
export const getProcessor = (event: LambdaEvent): EventProcessor<LambdaEvent> => {
  if (isProxyEventALB(event)) return albProcessor       // → ALBProcessor
  if (isProxyEventV2(event)) return v2Processor          // → EventV2Processor
  if (isLatticeEventV2(event)) return latticeV2Processor // → LatticeV2Processor
  return v1Processor                                     // → EventV1Processor
}
```

`handle`/`streamHandle` (both confirmed production-alive, both re-exported
from `index.ts`) call `getProcessor(event)` directly, in the same file, at
lines 148 and 253. `getProcessor` in turn references the module-scope
processor instances, which reference the processor classes and the
`LambdaEvent` union's four member types. Every one of these 9 symbols is
therefore reachable from a confirmed-alive production export via a plain,
unambiguous, intra-file reference chain — yet all 9 are claimed
`test-only`, citing a *different* test entrypoint
(`runtime-tests/lambda/index.test.ts`) as the sole reacher. **Root cause
not confirmed** (same posture as the fastify anomaly — a concrete,
reproducible, real-file data point handed to a core-implementer rather
than an asserted mechanism), but the shape strongly suggests a gap between
how production-side symbol reachability is computed (possibly stopping at
the first re-exported symbol rather than continuing to trace what that
symbol's body calls intra-file) versus test-side (which does trace
transitively — confirmed, since the test-side walk correctly reaches all
4 processor classes through `getProcessor`'s body).

A spot-check of 4 *other* hono `export`/`test-only` claims outside this
cluster (`equal`, `sha1`, `detectors`, `createWSContext`) found **zero**
production usage anywhere in `src/` for any of them — consistent with M4's
established "public-API-shaped, correctly hedged" class, confirming the
bug is bounded to this one file/cluster, not systemic across the 35.

**hono total: 11 of 43 test-only-verdict claims (26%) are confirmed false
positives**, from two independent, traced root causes (wildcard-exports
dist/src mismatch; aws-lambda intra-file reachability gap). Combined with
fastify's 9/10 zombie-test false positives, **20 of 64 test-only-verdict
claims across the corpus (31%) are confirmed wrong** — all masked to
medium by pre-existing whole-package hazards, none surviving to the
high-confidence gate on this corpus, but the isolated wildcard-exports repro
proves the same claim shape reaches HIGH the moment that masking hazard
isn't present.

## The axios story: M3→M4→M5

Task requirement #3. M3 found **134 test-file false positives** on axios
(every `*.test.js` flagged plain `unused`, no test/production distinction
possible — M5's tier-2 feature didn't exist yet). M4's T3.6 fix suppressed
all of them via zero-config test-root recognition — correct for the time,
but a blunt instrument: a suppressed claim produces **zero** signal, so a
genuinely test-only utility and a genuinely dead file both vanished into
the same silent "alive" bucket. M5 is the first milestone able to tell
these apart and say something positive rather than just "not flagged."

**This run: axios produces 0 export/file `test-only` claims and 7 zombie
`test` claims** (all `tests/module/{esm,cjs}/tests/*.module.test.{js,cjs}`
— a dedicated "does axios still work when consumed via `require`/`import`/
`require().default`/TS typings" compatibility suite). Triaged all 7
(exhaustive, population under 20):

- **2/7 genuine, correct zombie signal**
  (`fixture-cleanup.module.test.{cjs,js}`): these test *only* the suite's
  own `cleanupTempFixture` helper (`fs.rmSync` edge cases) — zero axios
  reference anywhere, static or dynamic. Exactly the tier-2 feature working
  as designed: a real, previously-invisible signal ("this test exercises
  only test infrastructure") that M3/M4 could never produce.
- **5/7 not a bug, a new unmodelled hazard class.** The other 5
  (`typings.module.test.{js,cjs}`, `ts.module.test.js`,
  `ts-require{,-default}.module.test.cjs`) each read a *second* `.ts`
  helper file as raw text (`fs.readFileSync`, not an import), write it into
  a generated temp-project, then `tsc`+`node` it as a **subprocess**. Those
  helper files (`esm-index.ts`, `ts-require.ts`, etc.) do
  `import axios from 'axios'` / `require('axios')` — genuinely exercising
  axios's real public API, just via a mechanism (dynamic file read → spawn
  → compile+run) no static reference-graph analyzer, ours or knip's, can
  see. This is a legitimate new hazard-class candidate ("source read as
  data and executed out-of-process"), not a defect in the existing
  mechanism — flagged for the hazard registry, not a bug fix.

**So: is the M5 zombie-test signal on axios right?** 2/7 yes outright;
5/7 no, but for an honest, explainable, non-bug reason that both tools
share equally — **0/7 are wrong for a reason unique to `unused`** (contrast
with hono/fastify's 20 confirmed *bugs* above). CI-seconds estimate:
7 × 5s = **35s/run** — directionally real (the 2 true-positive zombies
alone justify ~10s of it), the other 25s attributable to the unmodelled
dynamic-subprocess class rather than to genuinely dead tests.

**Net axios narrative: M3's 134 false positives → M4's 134 silent
suppressions → M5's 7 explicit, mostly-correct, fully-explained
test-only/zombie claims.** The signal graduated from "wrong" to "silent"
to "differentiated and mostly right" across three milestones on the exact
same file set — the intended trajectory for the tier-2 feature.

## Differential vs knip (pinned 6.27.0, `--no-config-hints`) — the test-only gap

Task requirement #4. Ran knip fresh on hono, axios, fastify (`.tool-versions`
copied into each clone; same known vitest-config-load caveat as M3/M4,
degrades gracefully in every case). **Structural point first**: knip's
`--reporter json` issue schema has categories `files` / `exports` / `types`
/ `dependencies` / `devDependencies` / `unlisted` / `unresolved` /
`duplicates` / `enumMembers` / `classMembers` / `nsExports` / `nsTypes` —
**no `test` category, no zombie/CI-seconds concept anywhere in the
schema.** A knip "unused file" issue on a test file and on a production
file are structurally identical JSON shapes; there is no way to tell them
apart from the report alone. Three concrete examples, all from axios
(knip's own `files` issues, verbatim):

1. `tests/module/cjs/tests/fixture-cleanup.module.test.cjs` — **ours**:
   `test`/`test-only` (zombie, confirmed TRUE POSITIVE above, contributes
   to the CI-seconds estimate). **knip**: `{"file": "...", "files":
   [{"name": "..."}], "exports": [], "types": [], "dependencies": [] ...}`
   — a flat "unused file," indistinguishable in shape from a dead
   production file. A user acting on knip's output alone gets "delete
   this," with no signal that it's a test that structurally can't touch
   product code.
2. `tests/module/cjs/tests/ts-require.module.test.cjs` — same: **ours**
   flags it `test-only` (the dynamic-subprocess hazard-class candidate
   above); **knip** flags the identical file, same flat shape, same
   missing distinction — both tools share the underlying blind spot, but
   only ours labels the claim as test-adjacent at all.
3. `tests/module/cjs/tests/helpers/ts-require.ts` (the `.ts` file dynamically
   read-and-compiled, never statically imported by anything) — **ours**:
   **no claim at all** for this file (correctly: it's a genuine template
   source for the subprocess-compile pattern, and we don't discover a
   reference *against* it either way since a template read is not a
   reference-graph edge). **knip**: flags it as a flat unused file, same
   shape as examples 1–2 — no way to distinguish "this is dead" from "this
   is a compile-fixture that's genuinely exercised out-of-process."

## Recommended fixes, ranked

1. **Wildcard `exports` subpaths have no `dist/`→`src/` remap, unlike
   singular entrypoint targets.** Confirmed root cause:
   `seedWildcardExportEntrypoints` (`analyze.ts:487`) matches a wildcard
   target's literal prefix (e.g. `dist/utils/`) against discovered
   *source* files directly — no equivalent to
   `detectProductionEntrypointsWithDiagnostics`'s documented
   `dist/**`→`src/**` remap (`emit.ts`, T3.6). On any unbuilt clone where
   the wildcard target's directory differs from source (i.e. any package
   with a build step — the overwhelmingly common case, and the exact
   "clone before build" flow this tool targets), the wildcard mechanism
   silently seeds **zero** entrypoints. The one existing unit test
   (`analyze.test.ts:159`) doesn't catch this because its fixture's target
   (`"./*": "./src/*.js"`) points directly at source, masking the gap.
   **Proven HIGH-confidence-capable** via an isolated, zero-hazard repro
   (above) — this run's top-priority fix. Minimum fix: apply the same
   `dist/**`→`src/**` remap logic to `seedWildcardExportEntrypoints`'s
   prefix match that `detectProductionEntrypointsWithDiagnostics` already
   applies to singular targets.
2. **A regression fixture for the wildcard-exports dist/src gap**: a
   package with `"./sub/*": "./dist/sub/*.js"`, real source under
   `src/sub/*.ts`, a file reachable only via that wildcard subpath, and a
   test for it — proving both the file classification and the zombie-test
   cascade get fixed together.
3. **`src/adapter/aws-lambda/handler.ts`'s processor-family reachability
   gap (M4's "residual anomaly," now confirmed as false `test-only`
   claims on 9 symbols, not just silent under-claiming).** `handle`/
   `streamHandle` (confirmed alive) call `getProcessor` intra-file;
   `getProcessor` and everything it references get claimed `test-only`
   regardless. Root cause not confirmed despite the trace above — flagged
   for a core-implementer with graph instrumentation on this exact file/
   commit. Priority: high — this is now a confirmed *false verdict*
   (test-only claimed on alive code), not just a missed claim.
4. **Fastify's `require('..')`/`require('../..')`-to-package-root
   zombie-test false positives (9 files, new anomaly).** Directory-style
   require resolution works correctly in every isolated repro attempted;
   the bug only manifests at real-repo scale and could not be isolated in
   a minimal fixture despite four attempts. `test/register.test.js` (not
   flagged) vs `test/route-prefix.test.js` (flagged) — byte-identical
   require shape — is the cleanest reproducible A/B for a core-implementer
   to instrument against. Priority: high — 9 of fastify's 10 zombie claims
   are wrong, on real functional test coverage.
5. **New hazard-class candidate: "source read as text data and executed
   out-of-process."** Axios's 5 dynamic-subprocess-compile test files
   (not a bug — a documented, honest scope gap, same treatment M4 gave
   the MDX-liveness gap). Worth a hazard-registry entry so future runs can
   distinguish "known unmodelled dynamic pattern" from "unexplained
   anomaly" in the summary output, the same way `config-referenced-file`
   already does for string-literal config references.
6. **Lower priority, unchanged from M3/M4**: the manual-build-tooling/
   npm-script-invoked false-positive class continues to be the largest
   claim-volume item (all of fastify's 21 `file`/`unused`, most of hono's
   established file claims) but remains correctly hedged at medium, zero

## Fixes

1. **Wildcard `exports` dist→src remap** (recommended fix #1 — the sole
   proven HIGH-capable defect). `seedWildcardExportEntrypoints`
   (`frontends/ts/analyze.ts`) now applies the same `dist/**`→`src/**` remap
   the singular-target path already used (T3.6): when a wildcard target's
   literal prefix (`./dist/utils/`) matches no source file, it retries with
   the leading `dist/` rewritten to `src/`. Grouped **by subpath key**, so a
   subpath whose `import` condition resolves is not penalised for `types`/
   `require` conditions pointing at other unbuilt dirs. A wildcard subpath
   that still matches nothing after the remap now raises the existing
   `unresolvable-entrypoint-target` project hazard (whole-package medium
   cap) — **never a silent drop**. Verified on the isolated repro: the
   HIGH-confidence `file`/`test-only` claim on the public tested util, and
   its zombie test, are both gone (0 claims, 0 high); a genuinely-missing
   wildcard still caps to medium, never leaking HIGH.

2. **Intra-file reachability** (recommended fix #3, the aws-lambda cluster —
   turned out systemic, not one file). Root cause confirmed: the IR modelled
   **no** intra-file symbol→symbol references, so an export reached by a
   *named* import (not a whole-surface import) never propagated liveness to
   the siblings it uses. Fixed at the extract→IR→reachability layers: the
   shadow-aware AST walk (`extract.ts`) now records references from each
   top-level declaration to the module-scope siblings (exported **or**
   private) it uses; `emitIR` (2g) flattens that adjacency into export→export
   `static` edges, tracing through private intermediaries (`handle` →
   `getProcessor` → `const albProcessor = new ALBProcessor()` →
   `ALBProcessor`); reachability's `processSymbol` walks them, and `claims.ts`
   excludes self-file edges from the orphan test so a truly-dead file with
   internal references still reads `unused`, not `none`. Only ever *adds*
   reachability from an already-reached symbol — lowers false positives,
   cannot introduce one.

3. **Zombie hardening** (recommended fix #4, fastify — see diagnosis below).
   A test is now exempt from zombie classification (degrade toward alive)
   when its reach carries an `unresolvable-import` hazard **or** references
   the analyzed package by its own name (a self-name import resolving
   *external* — the resolver limitation the fastify hypothesis named). Either
   means the analyzer cannot see what production code the test really runs —
   so no wrong zombie can survive an unbuilt / self-referencing test shape.
