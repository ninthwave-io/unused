## hono — triage

**High: 0.** Nothing to exhaustively triage — but this is the *first* and
most important finding, not a clean bill of health (see "Worst finding"
below): the reason hono has zero high claims is a **whole-package medium
cap**, not an absence of would-be false positives.

**Root cause of the whole-package cap.** hono's `tsconfig.json` has a
`references` array to 9 sibling tsconfig projects
(`tsconfig.build.json`, `tsconfig.spec.json`, five `runtime-tests/*/tsconfig.json`,
`perf-measures/type-check/scripts/tsconfig.json`). The `tsconfig-project-references`
hazard (architecture §4 class) therefore caps **every** claim in the package
at medium: *"tsconfig `references` composes this project with sibling
projects that may consume its files across the project boundary
(whole-package cap, medium)."* Confirmed correct behaviour of the hazard as
designed (conservative, zero-FP-risk) — but it is masking a much larger,
unrelated defect (below) that would otherwise be producing HIGH-confidence
false positives across a large fraction of hono's public API.

### Worst finding: unbuilt `exports`-map entrypoints silently collapse to one fallback file

hono's `package.json` declares `main`, `module`, `types`, and 75 `exports`
subpaths — **every single target** points into `dist/` (e.g.
`./dist/helper/accepts/index.js`, `./dist/types/adapter/service-worker/index.d.ts`).
This clone, checked out at a release tag exactly as a real engineer would
after `git clone` and before `npm run build`, has no `dist/` at all.

Traced in `packages/unused/src/frontends/ts/emit.ts`
(`detectProductionEntrypoints`, lines 511–554): each `main`/`module`/`exports`/`bin`
target is resolved via the T2.2 resolver; a target is only added as an
entrypoint if the resolver outcome is `internal`/`internal-declaration`.
Since every one of hono's 75+ declared targets resolves to a non-existent
`dist/` file, **`hits.length === 0`** for the *entire* package, and the code
falls through to `FALLBACK_CANDIDATES` (`index.ts`, `src/index.ts`,
`index.js`, `src/index.js`), silently picking the first that exists:
`src/index.ts`. Verified: `src/index.ts` imports only `./hono` (the core
`Hono` class) and re-exports types — it does **not** transitively reach
`src/adapter/*`, `src/helper/*`, `src/middleware/*`, `src/client/*`, or
`src/jsx/*`, each of which is hono's real, intentional, independently
publishable subpath entrypoint (`hono/aws-lambda`, `hono/accepts`,
`hono/client`, …).

Net effect: the analyzer silently discards ~74 of 75 declared public
entrypoints and treats a large fraction of the package's real public API as
unreachable. No warning surfaces (the `NO_ENTRYPOINTS_WARNING` path only
fires when the fallback *also* fails — here it doesn't, so the CLI looks
clean). The only thing standing between this and ~150–200 confidently-wrong
HIGH "unused" verdicts on a major real-world library's public API is the
unrelated tsconfig-`references` cap happening to also apply here. A
structurally similar hono-shaped repo *without* tsconfig project references
would fail the "zero high-confidence FP" gate outright. Confirmed with a
targeted differential against knip (below): knip does **not** make this
mistake on the same file (`src/adapter/aws-lambda/handler.ts`) — it
correctly treats the file as reachable.

Verdict: **FALSE-POSITIVE class, confirmed**, affecting most `src/adapter/*`,
`src/helper/*`, `src/client/*` claims. Missed reference: hono's own
`package.json#exports` (75 subpaths), which *should* count as the public
API surface per the PRD §4 assumption ("declared entrypoints are the
complete public API") — the assumption is being violated in practice
whenever exports targets don't exist on disk.

### Root-cause detail: the dynamic-import scope bug

`examples/server.js:173` contains `import('file://' + path.join(__dirname, url + '.js'))`.
The extraction logic (`recordDynamicImport`, `extract.ts:89-113`) correctly
classifies this as `computed: true` (the argument isn't a plain string
literal) and records a `computed-dynamic-import` hazard with a `scopePrefix`
computed by `staticSpecifierPrefix` (`extract.ts:382-403`). For a
`BinaryExpression` with `+`, that function recurses into the **left**
operand only — here, the `Literal` `'file://'` — and returns `"file://"` as
the "static prefix." Downstream, the hazard's protective cap is scoped to
files whose project-relative path starts with that prefix. **No file in any
real project has a path starting with `"file://"`** (it's a URL scheme, not
a filesystem path fragment) — so the hazard fires, records a scope, and that
scope matches zero files, silently providing no protection at all to the
five `examples/*/server.js` files it should be capping. This is a
confirmed, reproducible, high-value bug: **`staticSpecifierPrefix` needs to
either not treat a bare left-literal of a string-concatenation as a
path-shaped prefix, or fall back to "no static prefix" (whole-package cap)
when the extracted string doesn't look like a relative path** — the latter
is strictly safer and cheaper to implement.
Code: `packages/unused/src/frontends/ts/extract.ts:399-401`.

## fastify — triage

**High: 0.** Same shape as hono: one hazard blankets the entire package.

**Root cause.** `test/scripts/validate-ecosystem-links.test.js:9-11`:
```js
const modulePath = require.resolve('../../scripts/validate-ecosystem-links')
delete require.cache[modulePath]
return require(modulePath)
```
`require(modulePath)` has a computed (non-literal, bare-identifier) argument.
`staticSpecifierPrefix` returns `null` for a plain `Identifier` (correctly —
there's no literal to extract), so per the hazard's documented fallback ("or
the importer's whole package when there is no static prefix") the
`computed-require` hazard caps **the entire fastify package** at medium.
This is the hazard working exactly as designed and documented — conservative,
zero-FP-risk, but very heavy on recall (100% of 235 claims capped by one
line in one test-utility file). Distinct, and arguably more defensible,
than hono's case: here the fallback is real "no static prefix," not a
misfire.

**A concrete, cheap recall improvement exists nonetheless**: the require's
argument (`modulePath`) is a `const` assigned one line above from
`require.resolve('../../scripts/validate-ecosystem-links')` — a plain
string literal. A simple same-scope const-alias trace (recognize
`const x = require.resolve(<literal>); ...; require(x)` in the same
function scope) would let the analyzer resolve this specific, very common
idiom (used for cache-busting requires in tests) without sacrificing any
precision — no false negative risk, since the literal is unambiguous.

## Recommended fixes, ranked

1. **Interim test-file recognition, ahead of full M5.** The single largest
   FP source by volume across all three repos (100% of axios's 134 test-file
   high FPs; the large majority of hono's and fastify's medium claims).
   Doesn't need full tier-2 test-only-liveness semantics — even a
   zero-config `*.test.*`/`*.spec.*`/`__tests__/` pattern treated as a
   "keep-alive, no-claim" root (not a verdict, just a suppression) would
   collapse the dominant FP class immediately, well ahead of M5's scheduled
   arrival. Given the M3 acceptance line is "zero confirmed high-confidence
   FPs surviving the gate," and axios alone has 134 high-confidence FPs from
   this one gap, this is the highest-priority fix.
2. **Unbuilt-`exports`-map entrypoint fallback.** When `main`/`module`/`exports`/`bin`
   targets don't resolve to real files (a repo analyzed pre-build — an
   extremely common real-world scenario for `npx`-style first runs), the
   current all-or-nothing fallback to a single `index.*` file silently
   discards the rest of a multi-subpath `exports` map, producing systemic
   over-flagging of legitimate public API on any library structured the way
   hono is. Minimum fix: surface a loud warning when N of M declared
   entrypoint targets fail to resolve, and refuse `high` confidence
   anywhere in the run when that fraction is large (the same "prove nothing
   rather than flag everything" posture the zero-entrypoints case already
   takes). Better fix: resolve `exports` targets against the source tree
   when the `dist/`-shaped target is absent (e.g. via the sibling `.d.ts`'s
   implied source path, or a configurable src/dist mapping).
3. **`staticSpecifierPrefix` treats non-path string literals as path
   prefixes.** Confirmed, reproducible, single-line root cause
   (`extract.ts:399-401`) that makes a `computed-dynamic-import` hazard
   silently protect zero files instead of falling back to the safe
   whole-package cap. Cheap, precise fix; recommend a regression fixture
   using axios's exact pattern (`'file://' + path.join(...)`).
4. **Investigate the aws-lambda reachability anomaly** (hono, "Open
   anomaly" above) — a concrete repro where two files appear reachable
   despite their only importers being confirmed-dead, contradicting the
   documented BFS-from-entrypoints algorithm. Needs core-implementer time
   with graph internals, not something this triage can resolve from CLI
   output alone.
5. **Lower priority, document rather than fix urgently:** npm-script-invoked
   files (`sandbox/`, `examples/`, `gulpfile.js`, docs sub-package scripts)
   and nested-sibling-package test directories (`tests/smoke/*`,
   `tests/module/*`, `benchmarks/utils/*`). Confirmed via the knip
   differential to be a shared, industry-wide gap (knip's own defaults miss
   these too) rather than a regression unique to us — real users will
   reach for `ignore` config for these paths. Worth a documented assumption-set
   caveat in v1 so it isn't a surprise on first run, but not gate-blocking.
